# ADR-0002: Rails pre-commit hook tolerates missing rubocop gem and excludes encrypted credentials

- **Status:** Accepted
- **Date:** 2026-05-24
- **Supersedes:** (none — extends ADR-0001)
- **Superseded by:** (none)

## Context

ADR-0001 made `.pre-commit-config.yaml` the single source of truth for fast
quality checks and asserted as an invariant that the **bootstrap commit must
succeed** regardless of project shape. The `tests/e2e_rails.sh` sandbox
introduced alongside this ADR validated that invariant against a real Rails
project and exposed a structural gap:

`langs/rails/.pre-commit-config.yaml` runs `bundle exec rubocop` as a `system`
hook. That command requires (a) `rubocop` to be declared in the project's
`Gemfile` and (b) `bundle install` to have already run. Two common scaffolds
violate one or both:

1. `rails new --minimal` — produces a Gemfile that does *not* include
   `rubocop-rails-omakase`. Even after `bundle install`, the hook keeps
   failing with `can't find executable rubocop`.
2. `rails new` (full) before `bundle install` — Gemfile lists
   `rubocop-rails-omakase` but the gem is not installed yet, so the first
   commit after `install.sh` aborts with the same error.

Either case turns the very first `git commit` after bootstrap into a hard
failure, which is the exact regression ADR-0001 was meant to rule out.

Running the e2e against a `rails new --minimal` sandbox surfaced a second,
adjacent failure on the same commit: `end-of-file-fixer` "fixed"
`config/credentials.yml.enc` (Rails' encrypted credentials file, a binary
blob that has no meaningful EOL semantics). That auto-fix likewise exited
the commit non-zero and would in fact corrupt the encrypted payload.

## Decision

Two hook-level adjustments to `langs/rails/.pre-commit-config.yaml`:

**1. rubocop is gated on `bundle list rubocop`** — runs only when the gem is
actually resolvable in the project's bundle. Otherwise the hook is a
graceful no-op:

```yaml
- id: rubocop
  name: rubocop
  entry: bash -c 'bundle list rubocop >/dev/null 2>&1 && bundle exec rubocop "$@" || exit 0' --
  language: system
  files: \.rb$
```

`bundle list rubocop` exits 0 only when the gem is in the resolved bundle
(i.e. declared in Gemfile *and* `bundle install`'d). When it does, rubocop
runs normally and gates the commit. When it does not, the hook exits 0 and
the commit proceeds.

**2. Top-level `exclude: '\.enc$'`** — Rails encrypted credentials are
binary blobs generated by Rails itself; whitespace, EOL, and formatting
hooks would corrupt them and have no meaningful work to do. The exclude
applies to every hook in the file (both the upstream pre-commit-hooks set
and the local rubocop entry), which matches the reality that no fast-check
hook should ever touch `.enc` files.

## Consequences

- Positive: the ADR-0001 bootstrap-commit invariant now holds for every
  Rails scaffold, including `rails new --minimal` and the pre-`bundle install`
  state of a full `rails new`.
- Positive: projects that want rubocop simply add it to their Gemfile and
  `bundle install`; the hook auto-engages with no further config change.
- Positive: `tests/e2e_rails.sh` flips from FAIL to PASS without changes to
  the test itself.
- Negative: base-files' own e2e suite no longer exercises rubocop's runtime
  behavior end-to-end. That is acceptable — verifying rubocop is rubocop's
  responsibility, not this template's. The template's job is to wire the
  hook correctly; the hook's wiring is now verified by both the static
  `tests/test_pre_commit_yamls.sh` and the dynamic e2e flow.
- Neutral: the same pattern is *not* applied to Python (`ruff`, `ruff-format`
  via `astral-sh/ruff-pre-commit`) or Go (`go-fmt`, `go-vet`, `go-mod-tidy`
  via `dnephin/pre-commit-golang`) because those hooks fetch their tools
  through pre-commit's own dependency system rather than the project bundle.
  They do not have an analogous "tool missing from project manifest" failure
  mode. The single exception in the Go overlay is the `golangci-lint` local
  hook, which `tests/e2e_go.sh` already opts out of via `SKIP=golangci-lint`
  for the same structural reason.

This ADR extends ADR-0001 rather than reversing it: the invariant is the
same, and this decision closes the last remaining edge case that violated it.
